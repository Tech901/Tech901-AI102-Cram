Core Azure Content Safety — 4 Harm Categories

CategoryAPI TermDescription
Hate and FairnessHateDiscriminatory language targeting identity groups (race, gender, religion, etc.)
SexualSexualSexual content, nudity, exploitation, solicitation
ViolenceViolencePhysical harm, weapons, terrorism, graphic depictions
Self-HarmSelfHarmSelf-injury, suicide, eating disorders, self-destructive behavior
Multi-label classification: Content can trigger multiple categories simultaneously. Each category gets its own independent severity score.

4 Severity Levels

LevelValueDescription
Safe0General/professional context, appropriate for most audiences
Low2Prejudiced views, stereotyping, low-intensity harmful content
Medium4Offensive, insulting, or harmful instructions with moderate detail
High6Explicit, severe harmful content; radicalization; detailed harmful instructions
Severity values are even numbers: 0, 2, 4, 6 — not 0-3 or 1-4. This is a common exam trick.

Key Content Filters in Azure OpenAI

Default behavior: Medium threshold for all 4 categories on both prompts AND completions. Blocks Medium + High severity; allows Safe + Low.

Configurable Threshold Levels

ThresholdWhat Gets BlockedStrictness
Low + Med + HighBlocks everything except Safe (0)Strictest
Med + HighBlocks Medium (4) and High (6)Default
High onlyBlocks only High (6)Lenient
No filtersNothing blockedRequires approval
Annotate onlyNothing blocked, annotations returnedRequires approval
Tip: Filters are configurable separately for prompts (input) and completions (output). You can be strict on input and lenient on output, or vice versa.

Prompt Shields — Two Attack Types

Attack TypeDescription
User Prompt AttacksUser directly tries to jailbreak: change system rules, mockup conversations, role-play exploits, encoding attacks (Base64, ROT13)
Document Attacks (XPIA)Third-party content (retrieved documents, emails, web pages) embeds malicious prompts that hijack the model
XPIA = Cross-domain Prompt Injection Attack. This targets RAG scenarios where retrieved documents contain hidden instructions. Prompt Shields detect these automatically when enabled.

Detection Groundedness Detection

Detects hallucinated or ungrounded content by comparing model output against provided source materials.

ParameterLimit
Grounding sources55,000 characters max
Query + completion text7,500 characters max
Correction feature: When enabled, returns a correctedText field that rewrites ungrounded claims to align with the source material.
Tip: Groundedness detection is OFF by default. You must explicitly enable it in your content filter configuration. Useful for RAG pipelines where factual accuracy matters.

Detection Protected Material Detection

Text Detection

Identifies known copyrighted or protected content: song lyrics, news articles, recipes, and other published text.

Code Detection

Detects code from public GitHub repositories (indexed through April 6, 2023).

ReturnsDetails

Both text and code protected material detection are ON by default. They run on output (completions) only.

Framework AI Governance Lifecycle (NIST AI RMF-aligned)

Microsoft's AI risk management follows four phases aligned to NIST AI Risk Management Framework:

#PhaseActivities
1GovernDefine roles & responsibilities; establish policies; adopt RAI Standard; create oversight boards
2MapRisk identification; red teaming; threat modeling; stakeholder impact assessment
3MeasureSafety evaluations; fairness metrics; benchmark testing; automated testing pipelines
4ManageDeploy with guardrails; monitor in production; incident response; feedback loops
Mnemonic: G-M-M-M — Govern, Map, Measure, Manage. The cycle is continuous, not one-time.
Tip: Red teaming falls under the Map phase (risk identification), not the Measure phase. Measure is about quantitative evaluation.

Ownership Shared Responsibility Model

Microsoft's Responsibilities

Infrastructure security and availability
Base model safety training and alignment
Security Development Lifecycle (SDL)
Abuse detection and monitoring at platform level
Default content filters on all Azure OpenAI deployments

Customer's Responsibilities

Application-level safety (system prompts, guardrails)
Prompt engineering and grounding strategies
Content filter configuration for your use case
Use case compliance (legal, ethical, industry-specific)
Monitoring and logging of AI interactions
End-user transparency (disclosing AI usage)
Key point: Even though Microsoft provides default content filters, the customer is responsible for configuring them appropriately and ensuring overall application safety. Microsoft provides the tools; you own the implementation.
